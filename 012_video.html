🏪 Ejercicio 2: Sistema de Ventas

🏪 Sistema de Ventas

📋 Enunciado

Contexto: Base de datos para apoyar la gestión de un sistema de ventas. La empresa necesita llevar control de proveedores, clientes, productos y ventas.

🏗️ Entidades Identificadas

PROVEEDORES
• ID_Proveedor (PK)
• Nombre
• Calle
• Número
• Comuna
• Ciudad
• Teléfono
• Web
CLIENTES
• ID_Cliente (PK)
• Nombre
• Calle
• Número
• Comuna
• Ciudad
PRODUCTOS
• ID_Producto (PK)
• Nombre
• Precio
• Stock
• ID_Proveedor (FK)
• ID_Categoría (FK)
CATEGORÍAS
• ID_Categoría (PK)
• Nombre
• Descripción
VENTAS
• ID_Venta (PK)
• Fecha
• Descuento
• Monto
• ID_Cliente (FK)
TELÉFONOS
• Número (PK)
• ID_Cliente (FK)
CONTIENEN
• ID_Producto (PK, FK)
• ID_Venta (PK, FK)
• Cantidad

🔗 Relaciones del Sistema

Proveedores PROVEEN Productos: 1:N (Un proveedor puede proveer varios productos)
Productos PERTENECEN A Categorías: N:1 (Varios productos pueden pertenecer a una categoría)
Clientes REALIZAN Ventas: 1:N (Un cliente puede realizar varias ventas)
Ventas CONTIENEN Productos: N:M (Una venta puede contener varios productos, un producto puede estar en varias ventas)
Clientes TIENEN Teléfonos: 1:N (Un cliente puede tener varios teléfonos)
💡 Nota Importante sobre el Diseño

En este ejercicio se crea una entidad débil "TELÉFONOS" porque un cliente puede tener varios números de contacto. La dirección se desglosa en componentes (calle, número, comuna, ciudad) para mayor granularidad en la información y facilitar consultas específicas por ubicación.

🔍 Puntos Clave del Análisis

🎯 Identificación de Entidades

  • Sustantivos principales: Buscar palabras que representen objetos o conceptos sobre los que necesitamos almacenar información
  • Campos únicos: Identificar atributos que puedan servir como claves primarias
  • Entidades débiles: Reconocer cuando una entidad depende de otra para su existencia

📊 Definición de Atributos

  • Campos calculados: Evitar almacenar datos que se puedan calcular a partir de otros
  • Tipos de datos: Elegir el tipo más apropiado (entero, decimal, string, booleano, etc.)
  • Normalización: Evitar duplicidad de información entre tablas

🔗 Establecimiento de Relaciones

  • Cardinalidad 1:N: El lado "1" transfiere su clave primaria al lado "N"
  • Cardinalidad N:M: Se crea una tabla intermedia con las claves primarias de ambas entidades
  • Integridad referencial: Considerar acciones al eliminar registros relacionados

✅ Recomendaciones Finales

📝 Proceso Recomendado

Leer completamente el enunciado antes de comenzar
Subrayar sustantivos y posibles entidades
Identificar atributos de cada entidad
Determinar claves primarias
Establecer relaciones y cardinalidades
Revisar el texto para verificar que no falta información
Convertir a modelo relacional
Documentar acciones referenciales y restricciones adicionales
⚠️ Errores Comunes a Evitar
  • Duplicar información entre tablas relacionadas
  • No identificar correctamente las relaciones N:M
  • Olvidar transferir claves foráneas en relaciones 1:N
  • Usar tipos de datos incorrectos
  • No considerar la granularidad adecuada de los datos

🎯 Buenas Prácticas

  • Verificación cruzada: Revisar que cada requisito del texto esté cubierto
  • Validación lógica: Comprobar que las relaciones tengan sentido
  • Optimización temprana: Considerar el rendimiento desde el diseño
  • Documentación completa: Mantener registro de decisiones de diseño

🔧 Herramientas de Validación

  • Lista de verificación: Checklist de todos los requisitos
  • Matriz de trazabilidad: Relación entre requisitos y elementos del modelo
  • Casos de prueba: Escenarios para validar el diseño
  • Revisión por pares: Validación con otros desarrolladores
