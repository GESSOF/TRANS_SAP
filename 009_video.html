VIDEO 9: MODELADO DE DATOS

Curso de Modelado en SAP HANA
Nota: Esta conferencia comenzará a grabarse. El modelado de datos es uno de los pasos más importantes al iniciar el desarrollo de cualquier proyecto, ya que constituye la estructura sobre la que se apoyará toda la aplicación.

1. INTRODUCCIÓN AL MODELADO DE DATOS

El modelado de datos representa el corazón de todo aplicativo que necesite almacenar y consultar datos. Si el diseño de nuestra base de datos es deficiente, todo lo que construyamos sobre ella será también deficiente. Por ello, es fundamental tomarse el tiempo necesario y utilizar las metodologías correctas, ya que posteriormente será más complicado corregir errores.

El modelado debe servir como una guía general, mostrando un esquema de alto nivel que sea totalmente independiente de la base de datos o metodología utilizada para implementarlo. En esta fase, únicamente nos preocupamos por la estructura de los datos, sin considerar aún la tecnología específica a utilizar.

Definición: Los modelos muestran la estructura lógica de la base de datos, incluidas las relaciones y limitaciones que determinan cómo se almacenan los datos y cómo se accede a ellos. Por lo general se implementan de forma gráfica para facilitar su comprensión.

2. EVOLUCIÓN HISTÓRICA DE LOS MODELOS DE DATOS

A lo largo de la historia de las bases de datos, se han implementado distintas tecnologías, dando lugar a diferentes modelos:

2.1 Modelo Jerárquico

Utiliza una estructura tipo árbol donde cada nodo puede tener únicamente un nodo padre. Esta estructura es familiar y se utiliza en ciertos objetos de base de datos actuales.

2.2 Modelo en Red

Similar al modelo jerárquico, pero con la diferencia fundamental de que cada nodo puede tener más de un nodo padre, lo que permite mayor flexibilidad en el manejo de la información.

2.3 Modelo Orientado a Objetos

Representa la información en forma de objetos de programación. Su principal ventaja es que no requiere realizar joins entre tablas, ya que puede recuperar objetos directamente mediante punteros, ofreciendo un rendimiento superior en ciertos entornos.

2.4 Modelo Relacional

Este modelo marcó un antes y un después en el mundo de las bases de datos. Se basa en la organización de datos en tablas relacionadas entre sí.

2.5 Modelo Transaccional

Permite llevar a cabo un gran número de transacciones cortas en línea, con procesado de consultas más rápido, manteniendo la integridad de datos en entornos de acceso múltiple.

3. COMPONENTES DEL MODELO RELACIONAL

3.1 Entidades

Entidades: Representación de un objeto existente en el mundo real. Se convertirán en las tablas de nuestro sistema de almacenaje.

3.2 Atributos

Atributos: Características que definen o identifican a una entidad. Se convertirán en las columnas que almacenarán los datos en las tablas.

3.3 Relaciones

Relaciones: Vínculo que define una dependencia entre varias entidades. Constituye la gran potencia de las bases de datos relacionales.

3.4 Cardinalidad

Cardinalidad: Especifica el funcionamiento de la relación, indicando si un registro de una tabla puede relacionarse con uno solo de otra tabla o con múltiples registros.

3.5 Claves

Claves: Atributo de una entidad que distingue cada registro del conjunto de forma unívoca. Principalmente se refiere a la clave primaria.

4. PROCESO DE DISEÑO DE BASE DE DATOS

El diseño de una base de datos es el proceso en el que determinamos la organización de la misma, incluidas su estructura, contenido e incluso las aplicaciones que se han de desarrollar. Se considera una disciplina estable con métodos y técnicas propios.

Importante: Los términos "base de datos" y "tabla" no son sinónimos. Una base de datos relacional almacena su información en una o más tablas relacionadas.

4.1 Pasos del Proceso de Diseño

  • Determinar la finalidad de la base de datos: Es fundamental conocer qué funcionalidad queremos que tenga para estructurar adecuadamente el almacenamiento de información.
  • Buscar y organizar la información necesaria: Tener la información almacenada de manera organizada, ocupando el menor espacio posible y con acceso sencillo.
  • Dividir la información en tablas: Regla básica de las bases de datos relacionales: la información debe poder representarse siempre en forma de tabla.
  • Convertir los elementos de información en columnas: Cada dato que queramos introducir o consultar será una columna de una tabla.
  • No incluir datos calculados: Evitar redundancia de datos que pueden obtenerse mediante cálculos de columnas existentes.
  • Almacenar la información en sus partes lógicas más pequeñas: Por ejemplo, separar nombre, primer apellido y segundo apellido en campos independientes para mayor flexibilidad.
  • Especificar claves principales: Toda tabla debe tener un campo clave que permita distinguir un registro de otro de manera única.
  • Definir relaciones entre las tablas: Identificar los campos de unión y cómo se relacionan para complementar información.
  • Ajustar el diseño: Adaptar el modelo según necesidades y carencias identificadas.
  • Aplicar las reglas de normalización: Proceso que se desarrollará en detalle con el modelo relacional.

5. EJEMPLOS PRÁCTICOS

5.1 Ejemplo de Datos Calculados - Información Personal

En lugar de tener una columna con "nombre completo", es preferible tener:

  • Campo: Nombre
  • Campo: Primer Apellido
  • Campo: Segundo Apellido

Esto permite mayor flexibilidad en el uso de los datos y evita redundancia.

5.2 Ejemplo de Datos Calculados - Facturación

En un detalle de compra:

  • Producto: Tijeras
  • Precio unitario: 5€
  • Cantidad: 3 unidades
  • Total: 15€ (DATO CALCULADO - no almacenar)

El total se calcula multiplicando precio × cantidad, por lo que no debe almacenarse como columna independiente.
