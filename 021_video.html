🌐 Introducción a SAP Web IDE for SAP HANA

🎯 Definición Oficial

SAP Web IDE for SAP HANA es un entorno de desarrollo integrado basado en web que se utiliza para el desarrollo de aplicaciones basadas en SAP HANA.

🏗️ Componentes que Maneja

  • Interfaces de usuario - Web y móviles
  • Lógica empresarial - Procedimientos y funciones
  • Modelos de datos extensos - Estructuras complejas de HANA

🔗 Integración con Infraestructura

Funciona junto con componentes clave del ecosistema HANA:

  • HDI (HANA Deployment Infrastructure) - Infraestructura de implementación
  • XS Advanced (XSA) - Plataforma de tiempo de ejecución
  • Contenedores HDI - Abstracción de esquemas
🚫 Incompatibilidad Importante

SAP Web IDE for SAP HANA NO funciona con el desarrollo XS Classic. Solo es compatible con XS Advanced (XSA).

🎯 Enfoque Principal

✅ Incluye

  • Desarrollo de aplicaciones
  • Modelado de datos
  • Creación de vistas
  • Programación SQL

❌ No Incluye

  • Administración de BD
  • Gestión de usuarios
  • Configuración sistema
  • Monitoreo performance

🔧 Para Administración

Para tareas administrativas de la base de datos se utiliza otro entorno llamado HANA Cockpit, que complementa las funcionalidades del Web IDE.

⚖️ Diferencias: SAP Web ID vs SAP Web IDE for SAP HANA

🚨 Confusión Común en Nomenclatura

Es importante recordar que SAP Web IDE for SAP HANA NO es lo mismo que SAP Web ID. Esta similitud en nombres puede llevar a confusión.

🔵 SAP Web ID

  • Para: Motor XS Classic
  • Uso: HANA 1.0 principalmente
  • Desarrollo: Aplicaciones XS
  • Estado: Legacy/Obsoleto
  • Interfaz: Web básica

🟢 SAP Web IDE for SAP HANA

  • Para: Motor XSA
  • Uso: HANA 2.0
  • Desarrollo: Aplicaciones XSA
  • Estado: Actual/Oficial
  • Interfaz: Web avanzada

🔄 Además: SAP Web IDE (sin "for SAP HANA")

Existe una tercera herramienta llamada simplemente SAP Web IDE que:

  • Es similar a una herramienta de cliente
  • Se utiliza para desarrollos de aplicaciones basadas en Fiori y SAP UI5
  • Es diferente a SAP Web IDE for SAP HANA
💭 Nomenclatura Problemática

La similitud en los nombres es problemática porque:

  • Abreviación común: "SAP Web IDE" se usa para ambos
  • Contexto crucial: Hay que deducir de cuál hablan
  • En exámenes: Especial cuidado con la nomenclatura exacta

🎯 Regla Práctica

  • En el día a día: "Web IDE" = SAP Web IDE for SAP HANA
  • En documentación oficial: Leer el nombre completo
  • En certificaciones: Verificar la nomenclatura exacta

🏗️ Arquitectura de 3 Capas

📊 Diagrama de Capas

SAP Web IDE for SAP HANA funciona con una arquitectura de tres capas claramente diferenciadas:

🖥️ CAPA CLIENTE

SAP Web IDE for SAP HANA

Entorno que vemos en pantalla

⚙️ CAPA SERVIDOR

XS Advanced (XSA)

Servidor de aplicaciones

🗄️ CAPA DATOS

SQL + HDI

Base de datos y catálogo

🗄️ Capa de Datos (Inferior)

La capa inferior maneja el acceso a los datos:

  • SQL - Se comunica con los datos almacenados en la base de datos
  • Estructura HDI - Accede al catálogo de objetos (vistas, tablas, procedimientos)

⚙️ Capa Intermedia (Servidor)

La capa intermedia contiene el servidor de aplicaciones:

  • XSA (XS Advanced) - Motor de aplicaciones
  • Ejecución de procedimientos - Y funciones
  • Funcionalidad de programación - Para aplicaciones
  • Consulta de información - Desde la base de datos hacia el usuario

🖥️ Capa de Cliente (Superior)

La capa superior es la interfaz de usuario:

  • SAP Web IDE for SAP HANA - Entorno visual
  • Interfaz en navegador - Lo que vemos en pantalla
  • Herramientas de desarrollo - Editores y asistentes
  • Punto de interacción - Con el desarrollador

🔄 Flujo de Trabajo

  1. Usuario interactúa con Web IDE (Capa Cliente)
  2. Solicitudes se procesan en XSA (Capa Servidor)
  3. Datos se obtienen via SQL/HDI (Capa Datos)
  4. Resultados se muestran en la interfaz (Capa Cliente)

👁️ Vistas Principales del Web IDE

🎯 Dos Vistas Principales

SAP Web IDE for SAP HANA tiene dos vistas principales entre las que puedes cambiar fácilmente:

💻 Vista de Desarrollo

  • Escritura de código - Desarrollo de aplicaciones
  • Modelado de datos - Creación de modelos
  • Combinación de perspectivas - Modeling + Development
  • Completado de código - Asistencia inteligente

  • Consola SQL - Ejecución directa
  • Gestión de conexiones - Múltiples BD
  • Visualización de datos - Tablas y vistas
  • Análisis de metadatos - Estructura de objetos

🔄 Cambio entre Vistas

El cambio entre vistas es simple y se realiza mediante:

  • Botones en la interfaz - Cambio con un clic
  • Principalmente cambia la parte izquierda de navegación
  • Contexto diferente para cada tipo de trabajo

💻 Vista de Desarrollo - Detalles

Esta vista se utiliza para:

  • Escribir código de desarrollo de aplicaciones
  • Modelado y desarrollo de modelos de datos
  • Gestión de proyectos y organización de archivos
  • Creación de objetos - Vistas, procedimientos, funciones

🎁 Funcionalidades Adicionales para Desarrolladores

  • Completado de código - Control + Espacio para sugerencias
  • Debugger - Análisis sentencia a sentencia en procedimientos

  • Asistencia de contenido - Sugerencias contextuales
⚠️ Limitación del Debugger

El debugger no es tan completo como en otros lenguajes de programación. Solo se puede implementar en procedimientos que se estén creando en el sistema.

📁 Creación de Proyectos

🚀 Inicio de Proyecto

Para comenzar con el modelado o desarrollo de aplicaciones, primero necesitas crear un proyecto de desarrollo que proporcionará las plantillas y estructuras necesarias.

📋 Proceso de Creación

  1. Clic derecho en Workspace → New → Project from Template
  2. Seleccionar tipo: Multi-Target Application Project
  3. Configurar nombre y descripción del proyecto
  4. Asignar espacio de implementación
  5. Finalizar creación del proyecto base

📄 Archivos y Extensiones

📝 Sistema Basado en Archivos

El desarrollo en Web IDE está completamente basado en archivos, donde cada objeto de origen es un archivo simple identificado con una extensión específica.

🔗 Extensiones y Editores

Cada tipo de archivo tiene su propia extensión, y el Web IDE sabe qué editor abrir según la extensión:

.hdbtable

Tablas - Editor de texto

.hdbcalculationview

Vistas calculadas - Editor gráfico

.hdbfunction

Funciones - Editor de código

.hdbprocedure

Procedimientos - Editor de código

.hdbview

Vistas simples - Editor de texto

.hdbindex

Índices - Editor de texto

.hdbsynonym

Sinónimos - Editor de texto

.hdbsequence

Secuencias - Editor de texto

🎯 Dos Formas de Crear Archivos

🔧 Database Artifacts

  • Asistente con opciones
  • Selección de tipo predefinido
  • Configuración guiada
  • Plantilla automática

📝 New File

  • Creación manual
  • Extensión escrita a mano
  • Contenido desde cero
  • Mayor control

💾 Guardado de Archivos

El sistema de guardado tiene varias opciones:

  • Botón de guardar individual - Para el archivo activo
  • Botón guardar todo - Para todas las pestañas abiertas
  • Ctrl + S - Atajo de teclado
  • Asterisco (*) - Indica archivos no guardados

🔨 Proceso de Build

Una vez escrito el código:

  1. Guardar el archivo - Ctrl + S
  2. Clic derecho → Build → Selected Files
  3. El sistema intenta crear el objeto en la BD
  4. Se muestran errores en consola inferior
  5. Corrección de errores - Si es necesario
🔍 Análisis de Errores

La consola inferior muestra:

  • Líneas en rojo - Errores encontrados
  • Primera línea de error - Suele ser la más importante
  • Descripción del problema - Sintaxis incorrecta, etc.
  • Proceso de build - Preparando, desplegando, verificando

📤 Exportación de Archivos

Los archivos se pueden exportar como:

  • Archivos de texto plano - Sin formato especial
  • Editables externamente - Con cualquier editor de texto
  • Transportables - Entre sistemas fácilmente
  • Compatibles con Git - Control de versiones

📦 Contenedores HDI

🎯 ¿Qué es un Contenedor HDI?

Un contenedor HDI, también llamado esquema contenedor o simplemente contenedor, es una caja lógica que se encuentra sobre el esquema físico de la base de datos.

💡 Idea Básica

Permitir que el desarrollo se desacople de los componentes físicos de almacenamiento de la base de datos. Los desarrolladores nunca deben preocuparse por el despliegue físico de la base de datos.

🏗️ Funcionamiento

Cuando se construye un contenedor por primera vez:

  1. Se genera automáticamente un esquema físico
  2. Se enlaza este esquema al contenedor
  3. Los desarrolladores referencian solo el contenedor
  4. Nunca especifican el esquema físico

🏠 Analogía de Construcción

Es como si estuviéramos construyendo una casa:

  • Cada desarrollador - Construye su habitación en el taller
  • Trabajo independiente - Sin acceso al resto de la casa
  • Una vez terminado - Se transporta y pega al conjunto
  • El HDI se encarga - De encajar todo en su sitio

🏷️ Desarrollo Sin Esquema

Este enfoque se llama "desarrollo sin esquema" porque:

  • Los desarrolladores no conocen el esquema físico
  • Se referencian contenedores - No esquemas
  • Abstracción completa - De la implementación física
  • Gestión automática - De nombres y ubicaciones

🔑 Componente Clave HDI

El contenedor HDI es un componente clave de la nueva Infraestructura de Implementación de HANA (HDI):

  • Solo se utiliza con XSA - No con XS Classic
  • Reemplaza sistemas anteriores - De despliegue
  • Gestión moderna - De dependencias y versiones

📊 Nombres Técnicos Largos

Cuando se crea un contenedor por primera vez:

  • Genera su propio esquema físico - Automáticamente
  • Con nombre técnico largo - Basado en proyecto y versión
  • Se pueden crear objetos directamente - En el contenedor
  • Se colocan físicamente - En el esquema generado

🎁 Beneficio Clave: Control de Versiones

Una gran ventaja es que diferentes versiones de la base de datos pueden coexistir en el mismo sistema SAP HANA:

  • Actualizar una aplicación - Manteniendo la anterior
  • Nueva versión - Con objetos de BD mejorados
  • Esquema diferente - Para objetos más recientes
  • Aplicación anterior - Continúa con su esquema
📈 Comparación con XS Classic

En el desarrollo XS Classic:

  • Todos los objetos se implementaban en un esquema fijo (SISBD)
  • Nunca se podían tener múltiples versiones de objetos
  • Imposible soportar diferentes versiones de aplicaciones
  • HDI soluciona estos problemas fundamentales

✅ Ventajas de los Contenedores HDI

  • Aislamiento completo - Entre proyectos
  • Control de versiones - Múltiples versiones coexistentes
  • Desarrollo simplificado - Sin preocupación por esquemas
  • Despliegue automático - Gestión transparente
  • Escalabilidad - Facilita crecimiento de proyectos
