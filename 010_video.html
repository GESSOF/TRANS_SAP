Introducción a los Diagramas Entidad-Relación

1. Introducción al Modelo Entidad-Relación

Definición: Un modelo entidad-relación es un tipo de diagrama de flujo que ilustra cómo las entidades (elementos de la realidad) se relacionan entre sí dentro de un sistema.

Se trata de una colección de herramientas conceptuales para la descripción de:

  • Datos
  • Relaciones entre los datos
  • Semántica de los datos
  • Restricciones de los datos
Importante: Pensar en estos diagramas como una simple representación esquemática de cómo se van a almacenar los datos en la base de datos, su distribución y sus características.

Características del Modelo Entidad-Relación

  • Modelo de alto nivel: No tiene en cuenta aspectos técnicos de implementación
  • Representación conceptual: Solo realiza la representación esquemática de los datos
  • Enfoque en eficiencia: Busca conseguir un sistema lo más eficiente posible

2. Modelo Relacional vs. Modelo Entidad-Relación

Modelo Entidad-Relación (Alto Nivel)

  • Representación conceptual
  • No considera aspectos técnicos
  • Primer paso en el diseño

Modelo Relacional (Nivel Inferior)

  • Colección de tablas
  • Presenta datos y relaciones
  • Segundo paso en el diseño
  • Considera aspectos técnicos de implementación
Nota importante: Un esquema entidad-relación no se puede implementar directamente en una base de datos. Para ello existe el modelo relacional, que convierte y normaliza el esquema entidad-relación en tablas con estructura y relaciones implementables.

3. Conceptos Fundamentales

3.1 Entidades

Entidad: Elemento del mundo real que es distinguible de todos los demás. Puede ser una persona, un árbol, un coche, una casa, un aula, etc.

Conjunto de entidades: Conjunto de entidades que comparten los mismos atributos.

Ejemplo:
  • Conjunto de entidades: Árboles
  • Entidad individual: Cada uno de los árboles específicos
  • Condición: Todos los árboles deben tener los mismos atributos

Tipos de Entidades

  • Entidades fuertes: Funcionan de forma totalmente independiente
  • Entidades débiles: Tienen dependencia de otras entidades (se representan con rectángulo de línea doble)

3.2 Atributos

Atributo: Describe propiedades de cada miembro del conjunto de entidades. Se convierten en las columnas de las tablas.

Tipos de Atributos

  1. Simples: No contienen otros atributos (ej: nombre)
  2. Compuestos: Incluyen otros atributos simples (ej: dirección postal)
  3. Multivalorados: Pueden tomar muchos valores para una misma entidad (ej: números de teléfono)
  4. Derivados/Calculados: Su valor se calcula a través de operaciones con otros atributos
  5. Nulos: Atributos para los que se desconoce su valor o no existe

3.3 Relaciones

Relación: Establece un punto de unión y coherencia entre los atributos de diferentes conjuntos de entidades.

Representan las formas en que se relacionan nuestras entidades. Ejemplo: En un concesionario, la relación "venta" conecta vendedores con coches.

3.4 Identificadores (Claves Primarias)

Identificador: Atributo que puede identificar un único registro dentro del conjunto de entidades de forma única.

Características de los Identificadores

  • No puede haber dos registros con el mismo identificador
  • Puede ser un identificador compuesto (varios atributos)
  • Todas las entidades deben tener al menos un identificador
  • Se representa subrayando el nombre del atributo

4. Representación Gráfica

Elementos Gráficos

  • Entidades: Rectángulo con el nombre en el interior
  • Atributos: Elipse con el nombre del atributo
  • Atributos multivalorados: Elipse doble
  • Identificadores: Elipse con nombre subrayado
  • Relaciones: Rombo con el nombre de la relación (generalmente un verbo)
  • Entidades débiles: Rectángulo doble

5. Cardinalidad de las Relaciones

La cardinalidad expresa de qué manera se relacionan las entidades entre sí.

5.1 Relación Uno a Uno (1:1)

Definición: Una entidad A se asocia con solamente una entidad B y viceversa.

Ejemplo: Un coche y su tarjeta de circulación, una persona y su número de DNI.

5.2 Relación Uno a Varios (1:N)

Definición: Un elemento de la entidad A se asocia con uno o más elementos de la entidad B, pero cada elemento de B solo se relaciona con un único elemento de A.

Ejemplo: Un vendedor puede vender varios coches, pero cada coche es vendido por un solo vendedor.

5.3 Relación Muchos a Muchos (N:M)

Definición: La entidad A se asocia con una o más entidades B y viceversa.

Ejemplo: Estudiantes y asignaturas. Un estudiante puede estar matriculado en varias asignaturas y una asignatura puede ser estudiada por varios estudiantes.

Importante: En las bases de datos relacionales no se puede implementar directamente la relación N:M. Se debe convertir en una nueva tabla relacionada a ambas entidades con relaciones 1:N.

6. Metodología de Desarrollo

Pasos para crear un Diagrama Entidad-Relación

  1. Partir de una descripción textual (conversación con cliente o documento escrito)
  2. Identificar sustantivos → Candidatos a entidades o atributos
  3. Identificar verbos → Posibles relaciones
  4. Analizar frases → Determinar cardinalidad de las relaciones
  5. Elaborar el diagrama → Representación gráfica
  6. Añadir documentación adicional → Lista de atributos y restricciones no representables

Ejemplo Práctico: Concesionario

Descripción: "Necesitamos crear una base de datos para llevar el control de nuestro concesionario. Queremos almacenar los datos de nuestros vehículos para facilitar la búsqueda del más adecuado para cada cliente. Necesitaríamos la matrícula, el color, la cilindrada, los caballos de vapor y el tipo de combustible. También queremos guardar los datos de nuestros vendedores: nombre, apellidos, DNI, dirección y teléfono. En las ventas, necesitaríamos saber qué coche se ha vendido, qué vendedor participó y el importe."

Análisis:

  • Entidades identificadas: Vehículos, Vendedores, Ventas
  • Atributos de Vehículos: matrícula, color, cilindrada, caballos de vapor, tipo de combustible
  • Atributos de Vendedores: nombre, apellidos, DNI, dirección, teléfono
  • Atributos de Ventas: coche, vendedor, importe
  • Relaciones: "registrar ventas", "ha vendido"

7. Restricciones

Restricciones: Condiciones que obligan al cumplimiento de ciertos comportamientos en la base de datos.

Las restricciones sirven para:

  • Mantener la integridad de los datos
  • Lograr una mejor organización
  • Establecer reglas de negocio

8. Resumen Final

Puntos Clave del Modelo Entidad-Relación

  • Representa la realidad a través de un esquema gráfico
  • Emplea terminologías de entidades, atributos y relaciones
  • Es un paso previo e indispensable para el modelo relacional
  • Permite una comprensión clara de los requisitos del sistema
  • Facilita la comunicación entre desarrolladores y clientes

Herramientas Disponibles
