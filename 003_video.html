🎯 La Gran Diferencia

En SAP HANA debemos considerar una diferencia fundamental a la hora de trabajar: el almacenamiento en columnas versus el almacenamiento en filas.

Esta decisión impactará significativamente en el rendimiento de las consultas y la eficiencia del sistema.

🔑 Puntos Clave para Certificación

  • Tipo de almacenamiento: Debe especificarse al crear tablas
  • Rendimiento: Cada tipo es óptimo para diferentes casos de uso
  • Conversión: Es posible cambiar de un tipo a otro posteriormente
  • Examen: Preguntas frecuentes sobre orientación de tablas

💭 Representación Mental vs Realidad

Cuando pensamos en una tabla, visualizamos filas y columnas organizadas. Sin embargo, en memoria la realidad es diferente: una memoria lineal de una sola dimensión donde los datos se almacenan secuencialmente.

🗃️ Tipos de Almacenamiento en SAP HANA

🔄 Flexibilidad de SAP HANA

SAP HANA admite ambos tipos de tablas, ya que maneja tanto aplicaciones transaccionales como analíticas en la misma base de datos.

Los desarrolladores pueden decidir qué tipo de tablas necesitan para admitir sus aplicaciones: orientadas a filas u orientadas a columnas.

📊 Almacenamiento en Filas

Óptimo para: Sistemas transaccionales

• Consultas de registro completo
• Operaciones CRUD frecuentes
• Acceso a múltiples columnas

📈 Almacenamiento en Columnas

Óptimo para: Sistemas analíticos

• Agregaciones y reportes
• Análisis de pocas columnas
• Procesamiento masivo de datos

⚠️ Decisión Crítica

Es fundamental tener claro cuál utilizar en cada caso, aunque decidirse no es difícil ya que su funcionalidad está muy definida según el propósito.

⚙️ Sistemas Transaccionales vs Analíticos

⚙️ Sistemas Transaccionales

La mayoría de las tablas de base de datos relacionales se basan en el almacenamiento orientado a filas, ya que se considera el diseño óptimo para un sistema transaccional.

🔄 ¿Qué es un Sistema Transaccional?

Cualquier sistema que se alimente sobre una base de datos donde hay que:

  • Introducir información
  • Modificar esa información
  • Borrarla si ya no es necesaria

En resumen: estamos trabajando esos datos activamente.

📊 Sistemas Analíticos

SAP HANA no solo maneja transacciones, sino que también está dedicado al análisis. En el caso del análisis, las tablas con orientación a columnas son más óptimas.

🎯 Características de los Sistemas Analíticos

  • Infocubos: Cubos de información para análisis
  • Datos numéricos: Tabla central con valores para operaciones
  • Contexto: Tablas auxiliares que aportan dimensionalidad
  • Agregaciones: Operaciones de suma, promedio, conteo, etc.

🎯 Decisión por Estructura

La elección entre filas y columnas depende de la estructura de datos y el tipo de operaciones que realizaremos sobre ellos:

📊 Uso Recomendado - Filas

  • Sistemas de gestión empresarial
  • Aplicaciones transaccionales
  • Consultas de registro completo
  • Operaciones CRUD frecuentes

📈 Uso Recomendado - Columnas

  • Business Intelligence
  • Análisis de datos
  • Reportes y dashboards
  • Agregaciones masivas

📊 Almacenamiento en Filas

⚡ Eficiencia del Almacenamiento en Filas

Este tipo de almacenamiento es realmente eficiente cuando queremos buscar todas las columnas de un registro, ya que los valores se encuentran de forma consecutiva.

🔍 Ejemplo de Consulta Completa

Mostrar todo el contenido de la tabla:

1️⃣ Primera Lectura

1, Daniel, Rico, 34 → Primera fila completa

2️⃣ Segunda Lectura

2, Federico, García, 25 → Segunda fila completa

3️⃣ Tercera Lectura

3, Manel, González, 42 → Tercera fila completa

✅ Ventajas del Almacenamiento en Filas

  • Consultas completas: Muy eficiente para obtener todas las columnas
  • Inserción de datos: Simplemente escribe el siguiente registro al final
  • Lectura secuencial: Los datos están organizados consecutivamente
  • Operaciones CRUD: Crear, leer, actualizar, eliminar registros completos

❌ Desventajas del Almacenamiento en Filas

  • Agregaciones costosas: Debe leer muchas casillas innecesarias
  • Análisis por columnas: Ineficiente para operaciones como sumas o promedios
  • Desperdicio de lecturas: Lee datos que no necesita para llegar al deseado

📈 Almacenamiento en Columnas

⚡ Eficiencia del Almacenamiento en Columnas

Con la orientación por columnas, si queremos realizar agregados como sumatorios, tendremos las entradas en el disco de forma consecutiva, encontrándolas mucho más rápido.

✅ Ventajas del Almacenamiento en Columnas

  • Agregaciones eficientes: Datos consecutivos para sumas, promedios, etc.
  • Análisis de BI: Óptimo para Business Intelligence
  • Memoria selectiva: Solo carga columnas necesarias
  • Sin índices adicionales: Reduce complejidad del sistema
  • Procesamiento paralelo: Cada núcleo puede trabajar una columna

❌ Desventajas del Almacenamiento en Columnas

  • Consultas completas costosas: Debe recopilar datos de múltiples ubicaciones
  • Inserción lenta: Debe reorganizar toda la información en disco
  • Reconstrucción de registros: Costoso obtener todas las columnas

📊 Características Especiales según Documentación

🎯 Aplicaciones Analíticas

Las tablas de almacenamiento en columnas son eficientes para aplicaciones analíticas donde:

  • Las solicitudes de conjuntos de datos no son predecibles
  • Suelen requerir solo un subconjunto de los datos
  • Por lo general, solo se requieren algunas columnas
  • Solo las columnas requeridas se cargan en memoria

🔧 Ventajas Técnicas

  • Memoria optimizada: Evita cargar columnas no utilizadas
  • Sin índices adicionales: Reduce complejidad del sistema
  • Procesamiento paralelo: Cada núcleo trabaja con una columna diferente

🗜️ Compresión de Diccionario

📦 Compresión Automática

Cuando creamos una tabla de almacenamiento en columnas, los datos se comprimen automáticamente. Esta compresión se denomina compresión de diccionario.

⚠️ Importante para Certificación

Esta compresión solo se realiza en tablas orientadas a columnas y en las tablas con orientación a filas no se da.

🎯 Objetivo de la Compresión

⚡
Más Datos en Caché CPU
📉
Menos Accesos a RAM
💾
Menos Accesos a Disco

Cuanto menos espacio ocupen los datos, más datos podemos tener en la caché de la CPU, que es la memoria más rápida y la que más interesa tener ocupada.

🔄 Funcionamiento de la Compresión

📈 A Mayor Repetición, Mayor Compresión

  • Elimina repetición: Almacena cada valor único una sola vez
  • Usa números enteros: Para representar valores originales
  • Referencias especiales: Vinculan diccionario con tabla real
  • Menos espacio: Más eficaz para escanear

🗂️ Esquema de Compresión de Diccionario

📊 Datos Originales

Madrid, Barcelona, Madrid, Sevilla, Barcelona, Madrid

🗂️ Diccionario + Referencias

Diccionario:
1→Madrid, 2→Barcelona, 3→Sevilla

Referencias:
1, 2, 1, 3, 2, 1

✅ Beneficios Adicionales de la Compresión

  • Copias de seguridad: Más rápidas debido a menos datos
  • Restauraciones: Proceso acelerado
  • Espacio en disco: Las copias ocupan menos espacio
  • Transferencias: Movimientos de datos más eficientes

📝 Consideraciones

La cantidad de reducción en el volumen de datos depende de los mismos. Cuanta más repetición haya en los valores de los datos, mayor será la compresión obtenida.

⚡ Procesamiento en Paralelo

🚀 Revolución del Hardware

Gracias a las novedades aparecidas en el hardware, especialmente los procesadores multinúcleo, podemos crear aplicaciones de respuesta instantánea al distribuir las tareas entre los núcleos.

⚙️ Distribución Automática

La carga de trabajo se distribuye automáticamente entre todos los núcleos y garantiza que todas las partes del hardware contribuyan al rendimiento.

💿 Limitaciones de Discos Duros Tradicionales

Las bases de datos tradicionales no aprovechaban esta funcionalidad principalmente debido a la ubicación de los datos en el disco duro:

❌ Problema de los Discos Magnéticos

  • Una sola aguja: Solo podía estar en un lugar en cada momento
  • Sin lecturas paralelas: Imposible acceder a múltiples ubicaciones simultáneamente
  • Limitación física: Hardware secuencial por naturaleza

✅ Solución con Memoria RAM

  • Acceso aleatorio: Cualquier posición accesible instantáneamente
  • Múltiples accesos: Varios núcleos pueden leer simultáneamente
  • Aprovechamiento multinúcleo: Paralelización real

📊 Optimización para Columnas

Las tablas con almacenamiento orientado a columnas tienen la peculiaridad de que cada columna cargada en memoria es procesada por un procesador distinto.

📈 Escalabilidad del Sistema

🔧 Mejoras Posibles

  • Más procesadores: Aumentar capacidad de procesamiento paralelo
  • Más memoria RAM: Más información disponible inmediatamente
  • Mejor rendimiento: Mayor velocidad de procesado
  • Escalabilidad horizontal: Crecimiento según necesidades

🎯 Clave para Tiempo Real

El procesamiento en paralelo es la clave para permitir el procesamiento en tiempo real. Utilizando tablas con almacenamiento en columnas y con un particionamiento y modelos de datos bien diseñados, podemos esperar un rendimiento de la base de datos excelente.

🔀 Particionamiento de Tablas

🔄 Particionamiento en Columnas

Las tablas orientadas a columnas tienen la posibilidad de particionarse, que básicamente significa que en lugar de tener una sola tabla, tendremos varias tablas dedicadas a guardar el mismo tipo de información.

⚡ Mayor Paralelización

Podemos procesar cada partición con un núcleo diferente, logrando aún mayor paralelización y creando la tierra de cultivo perfecta para el procesamiento en paralelo.

📊 Límites de Particionamiento

1,000
Hasta HANA 1.0 SPS09
16,000
Desde HANA 1.0 SPS10

Desde SAP HANA 1.0 SPS 10 (anterior incluso a HANA 2.0), este límite se amplió a 16,000 particiones por tabla de tipo columna. La limitación es prácticamente inexistente.

🎯 Tipos de Particionamiento

📊 Por Valores

Información distribuida según valores de una columna específica

🤖 Automático

La base de datos decide cómo repartir la información

🔄 Round Robin

Un registro para cada partición, manteniendo balance

⚠️ Consideraciones de Diseño

🎯 No es Solo Partir y Acelerar

Para obtener el máximo beneficio se requiere un pequeño estudio de los datos:

  • Tipo de datos: Qué información estamos almacenando
  • Patrón de consultas: Cómo vamos a explotar esos datos
  • Distribución adecuada: Qué tipo de partición usar
  • Optimización específica: Según caso de uso particular

❌ Particionamiento Inadecuado

Si el particionamiento no está hecho de forma adecuada, conseguiremos una pequeña aceleración, pero será muy inferior a la aceleración que podríamos conseguir si hiciésemos las cosas correctamente.

🏗️ Estrategia de Particionamiento

1️⃣ Análisis de Datos

Estudiar patrones y características de la información

2️⃣ Definir Estrategia

Elegir tipo de particionamiento según uso

3️⃣ Implementación

Crear particiones optimizadas

4️⃣ Monitoreo

Verificar rendimiento y ajustar si es necesario

💾 Memoria Extendida y Tablas Multialmacenadas

🌡️ Recordatorio: Niveles Dinámicos

Como hemos visto, las tablas se encuentran principalmente en memoria. Si una tabla aún no está cargada, se carga automáticamente. Sin embargo, tenemos también los niveles dinámicos para trabajar con la temperatura de los datos:

🔥 Datos Calientes

Memoria RAM - Más críticos, acceso rápido

❄️ Datos Fríos

Disco Duro - Menos críticos, más baratos

📁 Memoria Extendida

Desde SAP HANA SPS09 han incluido este nuevo componente. Se trata de una capa de almacenamiento basada en disco, como si añadimos otro disco duro de forma paralela a la base de datos HANA existente.

🎯 Características

  • Para tablas específicas: Que no requieren rendimiento tan óptimo
  • Lectura más lenta: Al estar basado en disco
  • Flexibilidad: Mover tablas entre memoria RAM y extendida
  • Optimización de costos: Balance entre rendimiento y precio

🔄 Movilidad de Datos

En todo momento podemos pasar las tablas que nos interesen a esta memoria extendida, o tablas que ya se encuentren en ella podemos pasarlas a la memoria RAM para cambiar la velocidad de acceso.

📊 Tablas Multialmacenadas

Este es un tipo de tablas especial que apareció con HANA 2.0. Estas tablas tienen la peculiaridad de que están particionadas entre la memoria RAM y la memoria extendida.

⚠️ Requisito Previo

Para poder crearlas necesitamos tener instalada la memoria extendida. Esto no viene de serie, tenemos que instalarlo aparte.

🏗️ Arquitectura de Tabla Multialmacenada

🚀 Particiones en RAM

Datos de acceso frecuente
Alta velocidad

💾 Particiones en Memoria Extendida

Datos de acceso esporádico
Velocidad moderada

Una sola tabla para el usuario, múltiples ubicaciones de almacenamiento

🎯 Ventajas de las Tablas Multialmacenadas

✅ Beneficios

  • Transparencia: El usuario ve una única tabla
  • Criterio personalizable: Nosotros especificamos la distribución
  • Acceso optimizado: Datos frecuentes en RAM, ocasionales en disco
  • Balance perfecto: Rendimiento vs costo de almacenamiento
  • Gestión automática: El sistema maneja la distribución según criterios

🔧 Funcionamiento

Habrá datos a los que podremos acceder con gran velocidad y rendimiento, mientras que contendrá otros datos que serán más lentos de recuperar. La misma tabla, el mismo almacenamiento, pero velocidad de acceso variable según nuestro interés.

🎯 Conclusión

Con esto terminamos la parte de almacenamiento en SAP HANA. Hemos cubierto los fundamentos esenciales sobre tipos de almacenamiento, compresión, procesamiento paralelo y las nuevas funcionalidades como memoria extendida y tablas multialmacenadas.
