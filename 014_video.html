📝 Metodología de Análisis de Texto 07:37

📝 Método Sistemático de Análisis

Para extraer información estructurada de textos de requisitos:

  1. Marcar sustantivos → Candidatos a entidades o atributos
  2. Marcar verbos → Candidatos a relaciones
  3. Analizar contexto → Determinar función de cada elemento
  4. Descartar elementos irrelevantes → Filtrar información no pertinente

🎯 Criterios de Decisión

📦 ¿Cuándo es una Entidad?

  • Representa un objeto del mundo real
  • Tiene múltiples atributos asociados
  • Es central para el sistema
  • Se relaciona con otras entidades

📊 ¿Cuándo es un Atributo?

  • Describe una propiedad específica
  • No se subdivide normalmente
  • Pertenece claramente a una entidad
  • Es un dato simple
⚠️ Advertencia:

No todos los sustantivos serán entidades y no todos los verbos serán relaciones. El análisis de contexto es fundamental para tomar decisiones correctas.

👨‍🎓 Ejemplo Práctico: Alumnos y Profesores 08:01

📄 Texto de Requisitos del Cliente:

"Los requisitos para esta base de datos son los siguientes. En nuestro colegio tenemos una serie de alumnos de los que queremos guardar su nombre completo, su dirección, teléfono y DNI. Del mismo modo, tenemos a nuestro profesorado que imparte clases a los alumnos, del cual también queremos guardar su nombre completo, dirección, teléfono, DNI y un registro de su sueldo."

🔍 Análisis Paso a Paso

1. Sustantivos Identificados:

  • base de datos - Descartado (no es parte del dominio)
  • colegio - Contexto general
  • alumnos - 🏆 ENTIDAD
  • nombre, dirección, teléfono, DNI - 📊 ATRIBUTOS
  • profesorado - 🏆 ENTIDAD
  • clases - Parte de la relación
  • sueldo - 📊 ATRIBUTO

2. Verbos Identificados:

  • son, tenemos, queremos, guardar - Descartados (proceso)
  • imparte - 🔗 RELACIÓN

🔗 Establecimiento de Relaciones 26:26

🔗 Análisis de la Relación: IMPARTE

Pregunta clave: ¿Cuál es la cardinalidad de esta relación?

🤔 Análisis de Cardinalidad

  • ¿Un alumno puede tener uno o más profesores? → SÍ (N)
  • ¿Un profesor puede tener uno o más alumnos? → SÍ (N)
  • Resultado: Relación N:M (muchos a muchos)

📐 Representación Visual de la Cardinalidad

⚠️ Consideración Importante

Las relaciones N:M requieren tratamiento especial en el modelo relacional. Se convertirán en una tabla intermedia con las claves primarias de ambas entidades.

🎯 Tipos de Cardinalidad Recordatorio

  • 1:1 - Uno a uno (ej: Persona - DNI)
  • 1:N - Uno a muchos (ej: Departamento - Empleados)
  • N:M - Muchos a muchos (ej: Alumnos - Profesores)

🎯 Criterios para Atributos en Relaciones

  • El atributo depende de ambas entidades
  • No puede asignarse exclusivamente a una entidad
  • Su valor cambia según la combinación específica
  • Representa información sobre la conexión misma

🏢 Concepto de Mandantes en SAP 38:14

🏢 Sistema de Mandantes en SAP

Definición: Distribución por clientes que permite separar información en la misma base de datos física.

🔧 Funcionamiento del Sistema

  • Cada usuario se loguea a un mandante específico
  • Solo tiene acceso a información de ese mandante
  • Las tablas incluyen un campo "mandante" en la clave primaria
  • Permite reutilizar las mismas tablas para diferentes organizaciones

📊 Ejemplo de Tabla con Mandantes

Mandante Usuario ID Nombre Empresa
001 1 Juan Pérez Empresa A
002 1 María García Empresa B
003 1 Carlos López Empresa C

📝 Nota: Aunque el Usuario ID es igual (1), son registros diferentes por tener distinto mandante.

💡 Ventajas del Sistema de Mandantes
  • Seguridad: Aislamiento total de datos entre organizaciones
  • Eficiencia: Una sola instalación para múltiples clientes
  • Mantenimiento: Gestión centralizada del sistema
  • Escalabilidad: Fácil incorporación de nuevos clientes

🎯 Implicaciones en el Modelado

En sistemas SAP, todas las entidades principales deben considerar el campo Mandante como parte de su clave primaria para garantizar la separación de datos.

🏪 Ejercicio Completo: Sistema Comercial 44:00

📄 Enunciado Completo:

"Una empresa vende productos a varios clientes. Se necesita conocer los datos personales de los clientes (nombre, apellidos, DNI, dirección, fecha de nacimiento). 🔑 Definición de Claves Primarias 55:57

🔑 Importancia de las Claves Primarias

Toda entidad necesita un atributo (o conjunto de atributos) que identifique cada elemento de forma única e inequívoca.

🎯 Opciones para Claves Primarias

🆔 Campo Natural Único

  • DNI - Para personas
  • NIF - Para empresas
  • Código de producto - Asignado por empresa
  • Matrícula - Para vehículos

🤖 Campo Artificial

  • ID - Número secuencial
  • Código interno - Generado automáticamente
  • UUID - Identificador único universal
  • Referencia - Sistema de numeración propio

📋 Ejemplos en el Sistema Comercial

Entidad Clave Primaria Tipo Justificación
PROVEEDORES NIF Natural Único por empresa, identificación legal
PRODUCTOS Código Artificial Asignado por la empresa para identificación
CLIENTES ID Artificial Campo agregado, secuencial automático
💡 Consideraciones para Elegir Claves Primarias
  • Estabilidad: No debe cambiar con el tiempo
  • Simplicidad: Preferir claves simples a compuestas
  • Significado: Los campos naturales aportan valor semántico
  • Rendimiento: Los campos artificiales suelen ser más eficientes

🔧 Clave Compuesta

Cuando ningún campo individual puede garantizar unicidad, se combinan múltiples campos:

Ejemplo: Tabla de Ventas

  • Cliente ID + Producto Código + Fecha
  • Permite múltiples compras del mismo producto en fechas diferentes
